# 긴 마크다운 문서 무중단 생성 규칙 (requirements / design / tasks 등)

`requirements.md`, `design.md`, `tasks.md`, Implementation Plan 등 **긴 `.md`
문서**를 만들 때, 출력 토큰 한도로 인해 내용이 잘리거나 도구 호출이 실패하는
문제를 방지하기 위한 **강제 규칙**입니다. 기존 `markdown-rules.md`를 대체/보강
합니다. 아래 규칙은 **선택이 아니라 의무**입니다.

## 목차
1. 핵심 원칙 (왜 실패하는가)
2. 정량적 청크 상한선 (가장 중요)
3. 표준 작성 절차 (Outline → 청크 반복)
4. "끝까지 완료" 진행 추적 루프
5. 매 청크 후 검증
6. 금지 사항 (Anti-patterns)
7. 실행 체크리스트

## 1. 핵심 원칙 (왜 실패하는가)

- 실패의 원인은 **한 번의 도구 호출(fs_write/fs_append)에 너무 많은 내용을 담기
  때문**입니다. "이 정도면 한 번에 되겠지"라는 판단이 곧 실패로 이어집니다.
- 따라서 **"적당히 나눈다"는 판단을 신뢰하지 않습니다.** 대신 아래의 **정량적
  상한선**을 기계적으로 지킵니다.
- **채팅에 전문을 먼저 출력하지 않습니다.** 채팅 출력 → 파일 저장은 같은 내용을
  두 번 생성하는 이중 작업이고, 그 과정에서도 잘림이 발생합니다. 처음부터 **작은
  청크로 파일에 직접** 기록합니다. (사용자가 "채팅으로 먼저 보여줘"라고 명시적으로
  요청한 경우에만 채팅 출력을 하되, 그때도 아래 청크 상한선을 그대로 적용합니다.)

## 2. 정량적 청크 상한선 (가장 중요)

**한 번의 `fs_write` 또는 `fs_append` 호출은 아래 상한선을 절대 넘기지 않습니다.**

- **최대 약 40줄** 또는 **최대 약 1,500자** 중 **먼저 도달하는 쪽**을 상한으로 함
- 코드 블록/표가 포함되면 상한을 더 낮춰 **약 25-30줄**로 끊음
- 한 청크가 상한을 넘길 것 같으면 **반드시 더 잘게 쪼갬**
- "남은 내용이 얼마 안 남았으니 한 번에 끝내자"는 판단 **금지** — 남은 양과 무관
  하게 항상 상한선을 지킴 (이것이 마지막 20%가 실패하는 주된 원인)

## 3. 표준 작성 절차 (Outline → 청크 반복)

1. **작성 전 계획 수립:** 문서에 들어갈 **모든 섹션 목록을 먼저 나열**하고, 각
   섹션을 몇 개의 청크로 쓸지 대략 정한다. (예: "총 6개 섹션, 예상 10개 청크")
2. **1단계 — Outline 생성:** `fs_write`로 **제목 + 목차(섹션 리스트)만** 담긴 파일을
   먼저 만든다. 이때 본문은 넣지 않는다.
3. **2단계 이후 — 청크 반복:** `fs_append`를 사용해 **한 번에 청크 하나씩만** 추가
   한다. 절대 여러 섹션을 한 호출에 몰아넣지 않는다.
4. **한 청크 = 하나의 하위 섹션**, 또는 긴 하위 섹션의 **일부**. 짧은 섹션이라도
   상한선 안에서만 합친다.
5. 문서가 **완전히 끝날 때까지** 3-4단계를 반복한다.

## 4. "끝까지 완료" 진행 추적 루프

마지막 20%에서 멈추는 것을 막기 위해, **진행 상태를 명시적으로 추적**한다.

- 각 청크를 쓴 뒤, 내부적으로 **"작성 완료 섹션 / 남은 섹션"**을 갱신한다.
- **남은 섹션이 하나라도 있으면 멈추지 않고** 다음 `fs_append`를 계속 호출한다.
- "여기까지 하고 나머지는 다음에"라는 식의 **부분 종료 금지.** 사용자가 중단을
  요청하지 않는 한, **목차의 모든 항목이 채워질 때까지** 자동으로 이어서 진행한다.
- 여러 번의 도구 호출이 필요하더라도 그것이 정상이며, 호출 횟수를 아끼려고
  청크를 키우지 않는다.

## 5. 매 청크 후 검증

- 문서를 다 쓴 뒤(또는 중간중간), **파일의 꼬리(tail)를 확인**해 마지막 청크가
  잘리지 않고 온전히 저장됐는지 점검한다. (예: `tail -n 20 파일명`)
- 마지막 줄이 문장 중간에서 끊겼거나 코드 블록/표가 닫히지 않았으면, **끊긴
  지점부터 이어서** 다시 `fs_append`로 보완한다.
- 목차의 모든 섹션이 실제 본문으로 채워졌는지 **목차와 본문을 대조**해 확인한다.

## 6. 금지 사항 (Anti-patterns)

- 🚫 전체 문서를 **단일 도구 호출**로 작성 (`fs_write` 한 번에 전문)
- 🚫 "남은 내용이 조금이니 한 번에 마무리" — 상한선 초과의 주범
- 🚫 청크 상한선(약 40줄 / 1,500자)을 넘기는 append
- 🚫 채팅에 전문을 먼저 뿌린 뒤 그대로 복사해 저장 (이중 생성 → 잘림 위험)
- 🚫 목차 항목을 남겨둔 채 응답 종료 (부분 완료)

## 7. 실행 체크리스트

문서 생성 시 아래를 순서대로 지켰는지 확인한다.

- [ ] 작성 전 전체 섹션 목록과 예상 청크 수를 계획했는가
- [ ] `fs_write`로 제목 + 목차만 담은 파일을 먼저 만들었는가
- [ ] 이후 모든 추가는 `fs_append`이며, 각 호출이 약 40줄 / 1,500자 이하인가
- [ ] 코드/표 포함 청크는 25-30줄로 더 잘게 끊었는가
- [ ] 목차의 모든 섹션이 채워질 때까지 멈추지 않고 이어갔는가
- [ ] 마지막에 tail을 확인해 잘림/미완결이 없음을 검증했는가
