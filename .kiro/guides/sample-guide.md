# (샘플) 통합 가이드 템플릿

> 이 문서는 `.kiro/guides/` 폴더에 두는 **통합 가이드의 형식과 사용법을 보여주는 샘플**입니다.
> 실제 내용은 없으며, 팀/프로젝트 상황에 맞게 복사해 채워 쓰세요.
>
> 사용법: 관련 작업을 할 때 채팅에서 `#sample-guide`로 이 문서를 불러 참고하거나,
> `.kiro/steering/guide-routing-rules.md`의 키워드 매핑에 등록해 자동으로 참고되게 합니다.

## 통합 가이드란

여러 번 반복되는 유형의 작업(예: 특정 API 패턴, 화면 변경 패턴, 마이그레이션 등)을
매번 처음부터 설명하지 않도록, **공통 방법론을 한 곳에 모아둔 참조 문서**입니다.

- 평소에는 로드하지 않도록 front-matter를 `inclusion: manual`로 둡니다.
- 실제 작업 시 `#guide-name`으로 직접 부르거나, 라우팅 규칙의 키워드로 자동 참고시킵니다.

## 권장 구성

아래 섹션 골격을 복사해 채우면 됩니다. 항목은 작업 성격에 맞게 가감하세요.

1. **핵심 원칙** — 이 유형 작업에서 항상 지켜야 할 규칙(짧고 단정적으로).
2. **패턴/계열 분류** — 하위 유형이 여러 개면 구분 기준과 각각의 특징.
3. **표준 계약/구조** — 입출력 형식, 계층 구조, 네이밍 등 합의된 규약.
4. **정확성 속성(Correctness Properties)** — 결과가 맞다고 볼 수 있는 검증 가능한 성질.
5. **테스트·검증 전략** — 어떻게 검증할지(자동/수동, PBT/예시 테스트 등).
6. **새 작업 정의 체크리스트** — 새 스펙/작업을 시작할 때 밟는 순서.

## 작성 팁

- **고유식별 정보는 넣지 않는다**: 회사/제품 고유 패키지명, 실제 화면 ID, 테이블·컬럼명,
  저장 프로시저명, 고객번호·계정 등은 일반화하거나 플레이스홀더로 대체합니다.
  (예: `com.example.app`, `ScreenA`, `sample_table`, `sp_getSample`, `CUST_0001`)
- **짧고 단정적으로**: 규칙은 "~한다/하지 않는다" 형태로 명확히 씁니다.
- **정확성 속성 중심**: 구현 세부보다 "무엇이 참이어야 하는가"를 남기면 재사용성이 높습니다.
- 완료된 실제 스펙 목록을 표로 남기고 싶다면 식별 정보 없이 유형/패턴 수준으로만 적습니다.
